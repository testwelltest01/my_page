# 11. 리액트 마스터로 가는 길: 실무 고급 패턴

리액트의 기초를 넘어, 실제 중대형 프로젝트를 효율적이고 안전하게 관리하기 위해 반드시 알아야 할 실무 패턴들을 다룹니다.

---

## 1. 데이터 전달의 고통: Prop Drilling 해결하기

### 😫 문제 상황: Prop Drilling
부모가 가진 데이터를 저 밑에 있는 증손자 컴포넌트에게 주려고, 중간에 있는 자식들이 쓰지도 않는 데이터를 단순히 전달만 하는 현상을 말합니다. 코드가 복잡해지고 유지보수가 지옥이 됩니다.

### 😎 해결책: 전역 상태 관리 (Context API & Zustand)
- **Context API**: 리액트 내장 기능입니다. "데이터 저장소"를 하나 만들고, 필요한 컴포넌트만 바로 빨대를 꽂아서 쓰는 방식입니다.
- **Zustand**: 요즘 실무에서 가장 핫한 라이브러리입니다. Context API보다 설정이 훨씬 간단하고 강력합니다.

**Zustand 예시**:
```javascript
// store.js (창고 만들기)
import { create } from 'zustand';

const useUserStore = create((set) => ({
  user: null,
  login: (userData) => set({ user: userData }),
}));

// MyComponent.jsx (빨대 꽂기)
const { user } = useUserStore(); // 전역 어디서든 바로 가져옴!
```

## 2. 코드의 중복을 막아라: Custom Hooks (사용자 정의 훅)

### 🧐 왜 쓰나요?
서버에서 데이터를 가져오고, 로딩 처리하고, 에러 처리하는 로직은 어느 페이지에서나 똑같습니다. 이걸 매번 복사/붙여넣기 할 순 없죠? 이 **'로직'**만 따로 빼서 함수로 만드는 것이 Custom Hook입니다.

**예시 (useFetch 훅 만들기)**:
```javascript
// hooks/useFetch.js
export function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch(url).then(res => res.json()).then(data => {
      setData(data);
      setLoading(false);
    });
  }, [url]);

  return { data, loading };
}

// App.jsx (조립하기)
const { data, loading } = useFetch('/api/users'); // 한 줄로 끝!
```

## 3. 성능 최적화의 기술: "불필요한 일 하지 않기"

리액트는 함수가 재실행될 때 내부 로직도 다 다시 계산합니다. 이걸 막는 3총사가 있습니다.

- **useMemo**: 복잡한 **계산 결과값**을 기억해 둡니다. (값이 안 바뀌었으면 연산 생략!)
- **useCallback**: **함수 자체**를 기억해 둡니다. (자식 컴포넌트에게 함수를 줄 때 성능 최적화용)
- **React.memo**: **컴포넌트 전체**를 기억합니다. 부모가 바뀌어도 내 Props가 그대로면 난 다시 안 그릴래! 하는 선언입니다.

> [!TIP]
> **성능 최적화는 '비용'입니다.** 모든 곳에 쓸 필요는 없으며, 실제로 느려졌을 때나 리스트가 수천 개일 때 적용하는 것이 좋습니다.

## 4. 스마트한 폼 관리: 입력값이 10개라면?

### ❌ 나쁜 예 (모든 걸 따로 만들기)
```javascript
const [name, setName] = useState('');
const [email, setEmail] = useState('');
const [phone, setPhone] = useState(''); // 입력창이 많아질수록 지옥...
```

### ✅ 좋은 예 (객체로 묶기)
```javascript
const [formData, setFormData] = useState({ name: '', email: '', phone: '' });

const handleChange = (e) => {
  const { name, value } = e.target;
  setFormData({ ...formData, [name]: value }); // 스프레드 연산자로 한 방에 처리!
};
```
더 전문적으로는 **React Hook Form**이라는 라이브러리를 사용해 유효성 검사까지 한꺼번에 처리합니다.

## 5. 보안과 환경 설정: 환경 변수 (.env)

서버 주소나 API Key를 코드에 직접 쓰면(`Hard Coding`) 보안에 치명적이고, 나중에 서버가 바뀌었을 때 일일이 코드를 뒤져야 합니다.

### 🛠️ 사용법
1. 프로젝트 루트에 `.env` 파일을 만듭니다.
2. (Vite 기준) 변수명 앞에 `VITE_`를 꼭 붙입니다.
   ```env
   VITE_SERVER_URL=https://api.myserver.com
   VITE_KAKAO_ID=abcdef12345
   ```
3. 코드에서 불러오기: `import.meta.env.VITE_SERVER_URL`

**장점**: 
- 코드를 깃허브에 올려도 비밀키는 노출되지 않습니다. (.gitignore에 추가 필수!)
- 테스트 서버와 실제 서버 주소를 파일 하나만 바꿔서 교체할 수 있습니다.

## 6. 리액트의 심장 박동: 비동기 데이터와 재렌더링 루프

사용자님이 질문하신 **"데이터가 1초 뒤에 오면 리액트는 어떻게 다시 그리나요?"**에 대한 메커니즘을 아주 깊게 파헤쳐 봅시다. 이 과정을 이해하면 리액트의 모든 동작이 눈에 보이기 시작합니다.

### 🎢 리액트의 '렌더링 롤러코스터' 시나리오

#### 1단계: 컴포넌트의 출생 (Mounting & 1st Render)
- **상태 초기화**: `const [data, setData] = useState(null);` 호이스팅(준비) 단계에서 `data`는 `null`로 설정됩니다.
- **첫 번째 페인트**: 함수가 실행됩니다. `data`가 `null`이므로 `{data?.name}`은 아무것도 그리지 못합니다. 사용자에게는 "로딩 중..."이나 빈 화면이 보입니다.
- **예약**: `useEffect`가 실행되면서 "서버에 데이터 가져와!"라는 비동기 작업을 백그라운드에 던져놓습니다.

#### 2단계: 1초간의 정적 (Asynchronous Waiting)
- **놀랍게도**: 이 1초 동안 컴포넌트 함수는 **완전히 종료**된 상태입니다. 자바스크립트는 다른 일을 하고 있고, 화면은 첫 번째 렌더링 결과를 유지합니다.
- **주소 보존**: 리액트는 이 컴포넌트가 사용하던 `data`와 `setData`의 주소를 메모리에 소중히 간직하고 있습니다.

#### 3단계: 사건 발생 (Event Handling)
- **응답 도착**: 1초 뒤 서버에서 데이터가 도착합니다.
- **신호탄**: 비동기 콜백 함수 안에서 `setData(새데이터)`가 호출됩니다.

#### 4단계: 재탄생 (Re-rendering & 2nd Render) ⭐ 핵심!
- **트리거**: 리액트는 `setData`가 불리는 순간, "해당 컴포넌트가 더러워졌다(`Dirty`)"고 판단하고 다시 그리기를 스케줄링합니다.
- **함수 재실행**: 리액트 엔진이 **해당 컴포넌트 함수를 처음부터 끝까지 다시 읽습니다.**
- **상태 주입**: 이때 `useState`는 아까 저장해둔 `새데이터`를 꺼내서 `data` 변수에 꽂아줍니다.
- **최종 결과**: 이번엔 `data`에 값이 있으므로 `{data?.name}`이 실제 이름을 반환하고, 브라우저 화면에 정보가 업데이트됩니다.

### 🧠 기억해야 할 멘탈 모델
> **"리액트 컴포넌트는 애니메이션 프레임과 같습니다."**
> 
> 영화의 한 장면처럼, 값이 바뀔 때마다 리액트는 새로운 장면(Frame)을 찍어서 기존 장면 위에 겹쳐 보여줍니다. 이 속도가 너무 빨라서 우리는 화면이 부드럽게 변한다고 느끼는 것입니다.

### ⚠️ 주의할 점
이 '재실행' 과정에서 **함수 안에 선언된 일반 변수들**도 모두 초기화됩니다. 그래서 렌더링 사이에도 유지되어야 할 값은 반드시 `useState`나 `useRef`에 담아야 하는 것입니다!
